# 调试过程报告

本文件用于总结本项目在本轮开发与调试中的主要过程，并结合 Claude Code 历史聊天记录与后续接力排查结果，整理问题演化、修复路径和 AI 常见失误。

## 一、调试过程概览

本次调试大致可以分为四个阶段。

### 1. AI 难度体系重构

最早一轮工作集中在 AI 难度分层上，目标是把原先不够均衡的若干档 AI 改造成更清晰的五档体系：

- `easy`
- `medium`
- `hard`
- `expert`
- `joseki`

这部分的主要修改集中在：

- [src/game.cj](src/game.cj)
- [src/main.cj](src/main.cj)
- 前端难度入口相关文件

在这一阶段，Claude Code 曾尝试将中档难度统一到 Alpha-Beta 框架里，甚至一度想直接移除 `minimax`。随后用户明确指出：`medium` 不能简单删除 Minimax，而应保留 Minimax 作为中档难度的特征。这说明当时的实现更偏向“算法结构统一”，而不是“用户体验连续”。

### 2. 悔棋、复盘、帮助功能接续实现

根据 Claude Code 聊天记录，在余额耗尽之前，它已经为以下功能写出部分计划和半成品：

- 悔棋
- 终局复盘
- 帮助提示点

它在 [src/game.cj](src/game.cj) 中提前加入了部分历史记录字段，以及 `undo()`、`hint()` 相关逻辑，但没有把整个功能链条完全收口。之后继续接力时，又完成了：

- `GameState` 历史栈校正
- `/api/undo`
- `/api/hint`
- 前端按钮区
- 前端复盘状态
- 棋盘帮助高亮

这一阶段暴露出两个典型问题：

1. 悔棋后撤销的棋子没有直接消失，而是被前端误当成“新增白棋”做了落子动画。
2. 复盘层与终局结果层发生叠加，导致界面遮挡，操作困难。

这说明功能主线虽然接上了，但前端状态机和渲染逻辑没有一次性设计完整，而是边补功能边修状态冲突。

### 3. 前端文案与 UI 多轮返工

这一阶段的返工重点不在功能，而在表达。

从会话记录可以看出，用户多次指出以下问题：

- 文案有明显 AI 腔
- 栏目命名过于包装
- 算法说明不像写给玩家看的内容
- 页面里存在一些“即使语气正常，也没有存在必要”的废话

后续前端围绕以下文件进行了多轮修改：

- [frontend/index.html](frontend/index.html)
- [frontend/style.css](frontend/style.css)
- [frontend/app.js](frontend/app.js)

主要工作包括：

- 改写首页文案
- 改写游戏页说明文案
- 删除多余导语
- 重写算法说明区
- 清理高频“不是……而是……”式句型
- 调整结果卡、复盘区、状态提示区

这一阶段最大的调试成本，并不是技术实现，而是 AI 生成内容长期偏离真实产品语境，导致必须通过多轮人工纠偏才能落回可接受的产品表达。

### 4. 局域网联机模式补完

在本轮后期，项目又增加了“局域网联机”这一独立游戏模式。

后端方面，当前仓库中已经可以看到以下内容：

- [src/main.cj](src/main.cj) 中新增房间模型和房间数组
- 新增 LAN 相关接口：
  - `/api/lan/create`
  - `/api/lan/join`
  - `/api/lan/state`
  - `/api/lan/move`
  - `/api/lan/reset`
  - `/api/lan/leave`
- [src/game.cj](src/game.cj) 中新增 `playerMoveAs()`

前端方面，最初接手时 [frontend/app.js](frontend/app.js) 处于较危险状态，问题包括：

- 存在重复逻辑和残留分支
- 存在坏代码或截断代码
- 大量中文字符串已经乱码化
- 难以继续在局部上安全打补丁

因此后续处理方式不是继续局部修补，而是将 LAN 前端流程整体理顺，重建了以下能力：

- 首页“局域网联机”入口
- 创建房间
- 输入房间号加入房间
- 房间状态轮询
- 房主 / 客方身份展示
- 联机落子
- 房主重开
- 离房返回首页
- 终局后联机复盘

之后又继续补了两类体验修复：

1. 联机提示增强
   - 识别 `localhost/127.0.0.1` 不可分享
   - 显示当前访问地址
   - 支持复制地址
   - 支持复制房间号
   - 支持复制邀请链接

2. 加载层文案修复
   - 原来本地双人、联机对战也会显示“AI 正在思考”
   - 现已按模式区分为 AI 计算、联机同步或本地操作处理中

## 二、调试中的关键工程结论

结合整个过程，可以总结出以下几个关键判断。

### 1. 代码问题和环境问题必须分开看

在构建阶段，`cjpm build` 曾多次失败，但后续排查发现：

- 语义编译阶段其实已经通过
- 真正失败点常常是链接阶段
- 核心原因是 `target/release/bin/main.exe` 被运行中的进程占用

也就是说，当时真正的问题不是“代码编译不过”，而是“可执行文件被占用，导致链接失败”。如果不把这两个层次分开，很容易在错误方向上继续排查代码本身。

### 2. 前端状态问题不能只靠局部补丁解决

悔棋动画错误、复盘遮挡、LAN 前端残留碎片，这些问题的共同特点是：

- 单点功能已经“看起来能用”
- 但多个状态叠加后开始互相冲突

在这种情况下，继续沿着旧逻辑补丁式修修补补，风险很高。实际后续工作证明，适当整段重写状态流，比持续在脏逻辑上贴补丁更稳。

### 3. 文案也是调试的一部分

本项目里，前端文案的返工次数很多。虽然这看起来不像传统意义上的“bug 修复”，但它确实属于产品层面的调试：

- 页面传达的信息不对，也是一种功能失真
- 算法说明写成开发者自述，会直接影响用户理解
- 废话过多会让界面信息密度失衡

因此，这轮项目中“文案调试”并不是边角工作，而是主要返工来源之一。

## 三、AI 犯错总结

### 1. 容易优先追求结构统一，而不是用户体验连续

在 AI 难度重构阶段，最典型的问题就是试图把 `medium` 也统一成 Alpha-Beta 框架，从“实现结构更整齐”的角度看似合理，但忽略了用户明确希望保留 `minimax` 的差异体验。

这类错误的本质是：

- 把“实现层面的优雅”误当成“产品层面的正确”

### 2. 容易先补功能点，再事后修状态机

悔棋、复盘、帮助、联机都体现了同一个问题：

- 功能入口先加上
- 状态联动后补
- 复杂情况下再逐步修冲突

这样的实现路径短期推进很快，但很容易导致：

- 悔棋动画错误
- 弹层遮挡
- 轮询状态与本地状态冲突
- 真实局面与复盘局面混用

也就是说，AI 往往擅长“把功能点补齐”，但不一定会在第一轮就把完整状态流设计严谨。

### 3. 容易把文案写成“解释自己在设计什么”

这是本项目中最明显、也最频繁出现的问题之一。

典型表现包括：

- 栏目命名过度包装
- 高密度使用“不是……而是……”
- 解释页面设计意图，而不是解释给用户的信息
- 像作品展示页，而不是实际产品页

这说明 AI 很容易把“完整感”误写成“真实产品表达”，结果就是：

- 看起来很多字
- 实际信息含量不高
- 用户阅读成本上升

### 4. 容易忽略运行环境中的硬约束

例如：

- 可执行文件占用导致链接失败
- `localhost` 地址不能用于局域网分享
- 前端看到的“联机”其实依赖同一台服务器

这些问题并不是算法或语法错误，但却直接决定功能能否真正落地。AI 在实现阶段容易默认“环境配套是成立的”，直到用户指出后才开始回补提示或处理逻辑。

### 5. 在乱码环境下继续频繁改字符串，风险较高

当前仓库和终端输出里存在长期中文乱码问题。这样的环境会带来两个额外风险：

- 难以直接从终端确认字符串是否真实正确
- 修改文案时更容易把已有内容写坏

在这种情况下，如果 AI 仍然进行大量文案重写，而不先系统处理编码问题，后续返工概率会明显上升。

## 四、整体结论

本项目这轮调试最重要的经验，不是某一个具体 bug 如何修，而是以下三个判断标准：

1. 先判断这是代码逻辑问题，还是运行环境问题。
2. 先判断这是功能没实现，还是体验没闭环。
3. 先判断当前文件还能补，还是已经脏到应该整段重写。

如果把本次经验压缩成一句话，可以概括为：

> AI 最常见的失误不是“不会写代码”，而是“太快把事情写成一个看起来完整的答案”。

这种倾向在本项目中体现在：

- 难度设计
- 前端状态机
- 联机入口
- 文案表达
- 运行环境判断

后续如果继续迭代，建议始终保持以下策略：

- 产品规则优先于实现统一
- 状态流优先于功能点堆叠
- 环境约束优先于表面“编译通过”
- 文案按用户视角写，不按系统自述写
