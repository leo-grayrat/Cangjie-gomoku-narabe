# CLAUDE.md

## 项目定位

这是一个仓颉课程项目：使用华为仓颉语言开发网页版五子棋游戏。项目不能做成纯前端，必须有仓颉后端；前端只负责交互展示，核心游戏状态、胜负判断、AI 决策和必要接口应由仓颉后端承担。

## 工作方式

* 大改动前先进入 Plan 模式：先阅读项目结构，再给出分阶段计划，不要直接改代码。
* 使用 `cjpm` 管理项目，优先保证项目能稳定 `cjpm build`。
* 增量实现、增量验证：每完成一个小阶段就编译或运行验证，不要一次性写完整项目后才测试。
* 仓颉语法、标准库、网络编程、并发和包管理不确定时，优先查阅仓颉官方文档：https://cangjie-lang.cn/docs
* 可以参考现有五子棋 AI 编程指导：https://github.com/lihongxun945/myblog/issues?q=is%3Aissue%20state%3Aopen%20label%3A%E4%BA%94%E5%AD%90%E6%A3%8BAI%E6%95%99%E7%A8%8B%E7%AC%AC%E4%BA%8C%E7%89%88
* 参考外部教程时要消化为本项目实现，不要直接照搬不适配仓颉或当前架构的代码。

## 推荐实现路线

1. 先实现最小可运行版本：

   * 棋盘状态表示
   * 落子合法性检查
   * 五连胜负判断
   * 本地双人对弈
   * 简单网页交互

2. 再实现人机对战：

   * 先做可用的启发式评分 AI
   * 再加入 minimax / alpha-beta 剪枝
   * 根据性能情况逐步加入候选点裁剪、迭代加深、Zobrist 哈希或算杀逻辑
   * 不要一开始就写过度复杂的搜索框架

3. 最后做体验优化：

   * 前端显示当前玩家、胜负结果、重新开始按钮
   * AI 思考时给出状态提示
   * 页面风格要清爽、有完成度，避免典型 AI 生成感的大渐变、大圆角卡片堆叠

## 架构约束

* 前端负责：棋盘渲染、点击落子、显示状态、调用后端接口。
* 后端负责：棋局状态、规则判断、AI 计算、接口响应。必须为仓颉项目。
* 不要把核心规则和 AI 全部塞进 JavaScript。
* 不要过度工程化；课程项目以可读、可运行、可讲清楚为优先。
* 文件结构尽量简单，避免无意义拆分。

## 验证要求

每次声称完成前，必须提供实际验证证据：

* 编译：`cjpm build`
* 运行：说明启动命令和访问地址
* 功能：至少验证双人落子、非法落子、胜负判断、重新开始、人机对战
* 如果有前端页面，优先使用浏览器或 Playwright MCP 进行交互测试

## 排错原则

* 编译失败时先处理第一条错误，不要同时乱改多个无关位置。
* 仓颉语法不确定时，不要凭其他语言经验硬猜。
* 如果 WebFetch 访问失败，可以改用 shell/curl 获取官方文档或参考资料。
* 记录关键踩坑和修复原因，避免重复犯错。

## Git 与提交

* 不要提交构建缓存、临时文件、`.cache`、`CMakeFiles`、`target`、大体积中间产物。
* 不要提交 API Key、令牌、个人路径。
* 如果依赖本机绝对路径，必须提醒这会在助教机器上失效，并改成更可复现的配置。
